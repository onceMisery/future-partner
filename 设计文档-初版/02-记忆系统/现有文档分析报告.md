  二、参考资料深度解读

  2.1 VCPToolBox TagMemo Wave Algorithm 的核心贡献

  TagMemo V4→V8.1 的演进揭示了几个关键洞见：

  - V6 的 LIF 脉冲扩散：将静态标签图变成动态激活网络，spike 传播使"触景生情"式联想成为可能
  - V7 的有向序数拓扑：引入内在残差（intrinsic residuals），解决了无向图无法表达时序因果的问题
  - V8 的测地线重排：利用 LIF 扩散阶段缓存的 accumulatedEnergy 场，将 KNN 空间距离与拓扑距离混合，解决多义词陷阱
  - 残差金字塔：Gram-Schmidt 正交化多级剥离，捕捉被宏观概念掩盖的微弱语义信号

  这些机制的本质是：向量空间不是平坦的，标签拓扑构成了语义引力场，检索是在引力场中的测地线游走。

  2.2 Seahorse 的核心贡献

  Seahorse 的价值在于提供了一个可工程落地的脑区分工模型：

  - Cortex（HNSW）：纯向量计算，不碰 Tag 拓扑
  - Synapse（LIF + Tag 共现）：纯图扩散，不做向量检索
  - Thalamus（Tide/Gram-Schmidt）：意图门控，弱信号捕获，不存状态
  - Hippocampus（SQLite WAL + mmap）：纯存储，不做计算
  - Cerebellum（后台任务）：梦境/压缩/重索引，不阻塞主路径

  这个单一职责边界是最终版方案插件化设计的直接来源。

  三、现有方案的不足与可改进点

  3.1 架构设计的局限性

  问题一：Core 与插件的边界模糊

  现有方案的 Rust trait 定义（MemoryStore、VectorIndex、TagGraph 等）是好的起点，但缺少一个明确的 Kernel Contract。根据
  FAP-1 原则，Core 只定义契约，插件只提供算法或适配，不参与安全判定。现有方案中 PolicyEngine trait
  被设计为可插拔插件，这违反了 Kernel Contract 原则——安全判定不能外包给插件。

  // 现有方案（有问题）：PolicyEngine 作为可替换插件
  pub trait PolicyEngine {
      fn authorize(&self, ctx: AuthContext, op: MemoryOp, scope: MemoryScope) -> Result<Decision>;
  }

  // 应该是：PolicyEngine 是 Core 的不可替换内核，插件只能注册策略规则
  // 插件提供 PolicyRule，Core 的 PolicyEngine 执行裁决

  问题二：L0 的生命周期管理缺失

  现有方案描述 L0 为"ring buffer + SQLite WAL"，但没有定义 L0 → L1 折叠的触发机制和边界条件。OCMS 总体设计文档给出了
  retain_score 评分公式，但最终版没有将其纳入。这导致 L0 何时折叠、折叠粒度如何控制，在实现层面是空白的。

  问题三：L2 写入门槛缺乏量化

  最终版只说"L2 不能被低质量对话直接污染，必须经过 L1 → Semantic Extractor → Confidence Scoring → Policy Check → L2
  Merge"，但没有给出置信度阈值和评分公式。OCMS 总体设计给出了：

  L2 写入条件 = 信息稳定性 > 0.6 AND 可复用性 > 0.5 AND 来源可信度 > 0.7 AND 非临时闲聊

  这个量化标准在最终版中丢失了。

  3.2 技术选型的优化空间

  问题一：Tide 算法定义不清

  最终版引入了 tide 检索模式（"Gram-Schmidt / 投影熵 / 弱信号"），但没有给出具体实现。Seahorse README 中 Tide 算法基于
  Gram-Schmidt 分解，其核心是将查询向量投影到已有记忆的正交基上，计算残差能量，用于捕捉被主题掩盖的弱信号。这个算法在最
  终版中只是一个名字，缺乏数学定义。

  问题二：TagGraph 的冷热分离策略缺失

  OCMS 总体设计明确提出"热图保存在内存，冷数据保存在 tag_edge 表，服务启动时加载 Top-N
  热边，后台周期性压缩低权重边"。最终版的 TagGraph trait 没有体现这个冷热分离策略，在大规模场景下会有内存压力。

  问题三：向量索引升级路径不完整

  最终版提到"双索引切换，禁止原地覆盖"，但没有给出 embedding 模型升级时的具体迁移策略。当 embedding 模型从 v1 升级到 v2
  时，旧向量和新向量不在同一空间，需要双读对比期，这个过渡期的查询路由逻辑缺失。

  3.3 可插拔性和扩展性方面的不足

  问题一：插件注册机制缺失

  现有方案定义了 trait，但没有定义插件的注册、发现、版本管理机制。一个完整的插件系统需要：

  // 缺失的部分
  pub struct PluginRegistry {
      embedding_providers: HashMap<String, Box<dyn EmbeddingProvider>>,
      vector_indexes: HashMap<String, Box<dyn VectorIndex>>,
      // ...
  }

  pub trait Plugin: Send + Sync {
      fn name(&self) -> &str;
      fn version(&self) -> &str;
      fn capabilities(&self) -> PluginCapabilities;
  }

  问题二：DreamWorker 的审批流程未定义

  最终版提到"梦境操作默认生成 proposal，不直接改长期记忆，高风险变更必须审批"，但没有定义审批流程的状态机、审批者身份验
  证、审批超时处理。这是一个安全关键路径，不能留白。

  问题三：多租户下的插件隔离

  现有方案的插件设计是全局单例的，但在多租户场景下，不同租户可能需要不同的 embedding 模型或
  reranker。插件需要支持租户级别的配置覆盖。

  3.4 性能和安全方面的考虑

  性能缺口：
  - LIF 扩散的计算复杂度是 O(hops × top_k_neighbors × |E|)，在大规模 Tag
  图上可能超时。现有方案没有定义降级策略（高级检索超时可降级 basic 在 Phase 3
  验收标准中提到，但没有给出超时阈值和降级触发机制）
  - 霰弹枪查询的并发度没有限制，在长对话场景下可能产生 N+1 个并发向量检索，需要并发上限和背压机制

  安全缺口：
  - Prompt Injection 防护缺失：记忆写入路径上没有对内容进行 prompt injection 检测。恶意内容写入 L2 后，在检索时可能污染
  LLM 的推理
  - 向量反转攻击：没有对 embedding 输入进行长度和内容校验，超长输入可能导致 embedding 模型行为异常
  - 审计链的完整性验证：定义了哈希链结构，但没有定义链完整性的定期验证机制和断链检测

  3.5 与 FAP-1 协议的集成程度

  现有方案的 FAP-1 集成是协议层面的集成，但缺少语义层面的集成：

  - Intent Mandate 的 purpose 字段是自由文本，没有标准化的 purpose 词汇表，导致策略引擎无法做精确的 purpose-bound
  访问控制
  - ContextGrant 的 redaction_policy 是一个 ID 引用，但脱敏策略的定义格式、执行引擎、验证机制都没有定义
  - HandoffPacket 的 sbu_redaction_report 字段存在但没有定义其内容格式，接收方无法验证脱敏是否完整

  四、面向未来的可插拔记忆系统实现方案

  4.1 核心设计原则（对齐 FAP-1 Kernel Contract）

  原则一：Core 定义不可替换的 Kernel Contract- 安全判定（PolicyKernel）不可插拔
    - 审计写入（AuditKernel）不可插拔
    - 租户隔离（TenantKernel）不可插拔

  原则二：所有算法扩展点必须是插件
    - EmbeddingProvider（向量化算法）
    - VectorIndex（索引实现）
    - TagGraph（拓扑实现）
    - Reranker（重排算法）
    - DreamWorker（梦境算法）
    - ForgetEngine（遗忘算法）
    - CompactStrategy（折叠算法）

  原则三：插件只提供算法，不参与安全判定
    - 插件的输入输出经过 Core 的 PolicyKernel 过滤
    - 插件不能直接访问原始 MemoryUnit，只能访问 PolicyKernel 授权后的视图

  4.4 L0 折叠触发机制（补全缺失设计）

  pub struct L0FoldTrigger {
      token_threshold: usize,      // 默认 4096 tokens
      turn_threshold: usize,       // 默认 10 轮
      semantic_boundary_score: f32, // 语义边界检测阈值 0.70
      time_threshold_secs: u64,    // 会话静默超时 300s
  }

  impl L0FoldTrigger {
      pub fn should_fold(&self, state: &L0State) -> FoldDecision {
          // 优先级：强制折叠 > 语义边界 > token 阈值 > 轮次阈值
          if state.total_tokens > self.token_threshold * 2 {
              return FoldDecision::ForceFold;
          }
          if state.turn_count >= self.turn_threshold {
              return FoldDecision::Fold;
          }
          if state.semantic_boundary_score > self.semantic_boundary_score {
              return FoldDecision::FoldAtBoundary;
          }
          FoldDecision::Continue
      }
  }

  // retain_score 量化（补全 OCMS 总体设计中的公式）
  pub fn retain_score(event: &L0Event) -> f32 {
      0.30 * event.salience
      + 0.25 * event.stability
      + 0.20 * event.user_preference_signal
      + 0.15 * event.task_reuse_value
      + 0.10 * event.source_confidence
      - 0.20 * event.privacy_risk
      - 0.15 * event.noise_score
  }

  // L2 写入门槛（补全缺失的量化标准）
  pub fn l2_eligible(fact: &ExtractedFact) -> bool {
      fact.stability > 0.6
      && fact.reusability > 0.5
      && fact.source_confidence > 0.7
      && !fact.is_ephemeral_chat
  }

  4.5 安全增强：Prompt Injection 防护

  /// 记忆写入路径上的内容安全检查（Core 内置，不可绕过）
  pub struct ContentSafetyGuard {
      injection_patterns: Vec<Regex>,
      max_content_length: usize,
      embedding_input_max_tokens: usize,
  }

  impl ContentSafetyGuard {
      pub fn check_write(&self, unit: &MemoryUnit) -> Result<(), ContentSafetyError> {
          // 1. 长度检查
          if unit.content_text.len() > self.max_content_length {
              return Err(ContentSafetyError::TooLong);
          }
          // 2. Prompt injection 模式检测
          for pattern in &self.injection_patterns {
              if pattern.is_match(&unit.content_text) {
                  return Err(ContentSafetyError::InjectionDetected);
              }
          }
          // 3. SBU 自动标记（不阻止写入，但标记）
          Ok(())
      }
  }

      │                         │
      │                    [审批拒绝]──→ REJECTED
      │                         │
      │                    [超时 72h]──→ EXPIRED
      │
      └─[SBU 内容检测]──→ BLOCKED（不可审批通过）

  风险评级规则：
  - 涉及 L2 Fact 删除或降权 → 高风险
  - 涉及实体合并 → 中风险，需置信度 > 0.85
  - 涉及 L1 压缩 → 低风险，可自动审批
  - 涉及任何 SBU 标记内容 → 强制阻止

  4.8 检索降级策略（补全性能缺口）

  pub struct RetrievalFallbackChain {
      timeout_ms: u64,  // 默认 200ms
  }

  impl RetrievalFallbackChain {
      pub async fn retrieve(&self, req: &RetrieveRequest) -> RetrieveResult {
          match req.mode {
              RetrievalMode::TagmemoGeodetic => {
                  // 尝试完整 tagmemo + geodesic，超时降级
                  timeout(self.timeout_ms, self.tagmemo_geodesic(req))    .await
                      .unwrap_or_else(|_| {
                          // 降级到 hybrid
                          self.hybrid(req)
                      })
              }
              RetrievalMode::Tagmemo => {
                  timeout(self.timeout_ms, self.tagmemo(req))
                      .await
                      .unwrap_or_else(|_| self.basic(req))
              }
              _ => self.execute_direct(req).await,
          }
      }
  }
