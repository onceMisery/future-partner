下面按“**可直接落成正式设计文档**”的方式，基于前面确定的 FAP-1 方案进行逐章节详细设计。
这里的设计目标是：**从 0 到 1 实现一个独立 FAP 协议，不依赖 VCPToolBox，但吸收其高性能思想**。VCPToolBox 可作为启发来源，例如它的插件生命周期、分布式工具执行、SQLite + Rust 向量索引、WebSocket 分布式节点等架构实践；但 FAP 必须重新定义协议、消息、运行时、安全和扩展边界。VCPToolBox 架构文档显示其当前核心是 Node.js + PluginManager + WebSocketServer + KnowledgeBaseManager，并通过插件驱动、分布式工具注册、SQLite、Rust 向量索引等实现能力增强。 你的 FAP 文档则明确建议收敛为 Agent Card 发现、会话协商、Protobuf over QUIC/HTTP3、组织内 mTLS/JWT、跨组织 DID/VC 可选、潜空间仅实验能力。

# 第 1 章：设计目标与非目标

## 1.1 设计目标

FAP-1 的目标是提供一个面向未来 Agent 系统的通用通信协议，覆盖：

```text id="n3z3m5"
Agent 发现
Agent 会话建立
能力声明与绑定
安全认证与授权
工具/任务调用
多模态数据传输
插件运行时
分布式节点协作
高风险动作审计
协议扩展与演进
```

核心目标：

| 目标   | 说明                              |
| ---- | ------------------------------- |
| 独立协议 | 不依赖 MCP、A2A、VCPToolBox，可桥接但不绑定  |
| 高性能  | 支持 QUIC、多路复用、批量调用、异步执行、零拷贝数据面   |
| 可插拔  | 所有能力、传输、认证、检索、桥接、研究能力均插件化       |
| 安全可控 | 高风险动作必须 mandate + receipt       |
| 可审计  | 关键调用必须生成可验证 Receipt             |
| 可迁移  | 可从 MCP/A2A/VCP 风格工具调用迁移         |
| 可演进  | 潜空间、Gossip、CRDT、DID/VC 等进入扩展路线图 |

## 1.2 非目标

第一版不追求：

```text id="qq5iai"
不默认启用潜空间通信
不默认启用 Gossip 网络
不默认启用 CRDT 协同状态
不默认启用全量 DID/VC
不要求所有插件常驻进程
不强制所有模型使用 JSON Function Calling
不复制 VCPToolBox 的实现结构
```

## 1.3 协议分层定位

```text id="h3cb5p"
FAP Core:
  稳定、安全、强类型、可审计。

FAP-HP:
  高性能插件执行、动态工具注入、上下文折叠、分布式节点、批量调用。

FAP-EXT:
  潜空间、Gossip、CRDT、DID/VC、联邦学习等研究扩展。
```

# 第 5 章：会话层设计

## 5.1 会话状态机

```text id="vbnwuy"
NEW
  -> DISCOVERED
  -> HELLO_EXCHANGED
  -> AUTHENTICATED
  -> CAPABILITY_BOUND
  -> ACTIVE
  -> DRAINING
  -> CLOSED
```

异常状态：

```text id="acbm0a"
AUTH_FAILED
POLICY_REJECTED
EXPIRED
RESUMING
BROKEN
```

## 5.2 会话建立流程

```text id="ag2e89"
ClientHello:
  客户端声明版本、传输、profile、限制、nonce。

ServerHello:
  服务端选择版本、传输、profile、auth mode、session_id、nonce。

AuthInit:
  客户端提交 JWT 或 DID/VC proof。

AuthAck:
  服务端返回 subject、scope、granted capabilities。

CapabilityBind:
  客户端请求绑定能力。

CapabilityAck:
  服务端确认可用能力。
```

## 5.3 会话恢复

```text id="72b7mp"
客户端断线
  -> 重新连接
  -> ResumeRequest(session_id, last_received_sequence_no)
  -> 服务端检查会话是否仍有效
  -> ResumeAck(replay_from_sequence_no)
  -> 重放未确认 ProgressEvent / Result / Receipt
```

## 18.2 核心调用入口

```rust id="s2tp8h"
pub async fn handle_envelope(
    ctx: RequestContext,
    envelope: Envelope,
    services: &Services,
) -> Result<Envelope, FapError> {
    services.validator.validate_header(&envelope.header)?;
    services.replay.check_and_store(&envelope.header)?;
    services.security.verify(&ctx, &envelope).await?;
    services.policy.check(&ctx, &envelope).await?;

    match envelope.body {
        Some(envelope::Body::ClientHello(msg)) => {
            services.session.handle_client_hello(ctx, msg).await
        }
        Some(envelope::Body::AuthInit(msg)) => {
            services.security.handle_auth_init(ctx, msg).await
        }
        Some(envelope::Body::InvokeRequest(msg)) => {
            services.invocation.invoke(ctx, envelope.header, msg).await
        }
        Some(envelope::Body::BatchInvokeRequest(msg)) => {
            services.invocation.batch_invoke(ctx, envelope.header, msg).await
        }
        _ => Err(FapError::unsupported_message()),
    }
}
```

# 最终设计结论

这份章节级设计的核心是：

```text id="p87qwn"
FAP-1 Core:
  独立、安全、强类型、可审计的 Agent 通信协议。

FAP-HP:
  从 0 重新实现的高性能能力层，吸收 VCP 的插件化、动态注入、并行执行、分布式节点、统一文件解析思想。

FAP Plugin Runtime:
  支持 Rust crate、WASM、Java Sidecar、Python Sidecar、Remote Node。

FAP-EXT:
  潜空间、Gossip、CRDT、全量 DID/VC 的受控实验区。
```

一句话：**FAP Core 要稳定，FAP-HP 要高性能，FAP-EXT 要克制；所有能力都通过插件系统进入协议，而不是污染协议内核。**
