**重新从 0 到 1 设计一个完整 FAP 协议，只吸收 VCP 的高性能思想。**

VCPToolBox 的价值点主要是插件驱动、动态工具注入、分布式工具执行、SQLite + Rust 向量索引、上下文折叠、异步并行工具调用、统一文件访问等。它的架构文档显示其当前实现是 Node.js 核心，包含 PluginManager、WebSocketServer、KnowledgeBaseManager，支持插件生命周期、分布式工具注册与执行、SQLite 和 Rust N-API 向量索引等能力。
而你自己的 FAP 目标文档强调：最终应收敛为 FAP-1，采用 Agent Card 发现、MCP 式会话协商、Protobuf over QUIC/HTTP3、组织内 mTLS/JWT、跨组织 DID/VC 可选、潜空间只作为实验能力。

完整的 **FAP-1 从 0 到 1 设计方案**。

# 1. 协议定位

## 1.1 协议名称

```text id="jeynas"
FAP-1 = Future Agent Protocol 1
```

FAP-1 是一个面向 Agent-to-Agent、Agent-to-Tool、Agent-to-Memory 的原生通信协议。

它不是：

```text id="g5033q"
MCP 的复制品
A2A 的复制品
VCPToolBox 的移植版
某个工具调用格式
某个插件框架
```

它是：

```text id="y1ws2k"
发现协议
+ 会话协议
+ 能力协议
+ 安全授权协议
+ 数据传输协议
+ 审计协议
+ 插件运行时协议
```

# 2. 总体设计原则

## 2.1 核心原则

```text id="9zgprv"
1. FAP Core 保守稳定。
2. 高性能能力进入 FAP-HP Profile。
3. 研究性能力进入 Extension Profile。
4. 插件能力必须受 Core 安全链路约束。
5. 所有高风险动作必须可授权、可审计、可回滚。
6. 模型友好语法可以保留，但不能作为底层 wire protocol。
```

## 2.2 三层协议模型

```text id="awpyj8"
FAP Core:
  负责稳定通信、安全、会话、能力、数据、审计。

FAP-HP Profile:
  负责高性能插件执行、动态工具注入、上下文折叠、批量调用、分布式节点、统一文件解析。

FAP-EXT Profile:
  负责潜空间、Gossip、CRDT、全量 DID/VC、联邦学习等研究能力。
```

pub struct ExecutionScheduler {
    local_pool: LocalWorkerPool,
    warm_pool: WarmWorkerPool,
    remote_router: RemoteCapabilityRouter,
    idempotency: IdempotencyStore,
    receipt: ReceiptService,
}

impl ExecutionScheduler {
    pub async fn execute(&self, req: InvokeRequest) -> Result<InvocationResult, FapError> {
        let plan = self.plan(&req).await?;

        match plan.mode {
            ExecutionMode::LocalColdStart => self.local_pool.spawn_once(req).await,
            ExecutionMode::LocalWarmWorker => self.warm_pool.invoke(req).await,
            ExecutionMode::RemoteNode(node_id) => self.remote_router.forward(node_id, req).await,
            ExecutionMode::AsyncTask => self.spawn_async(req).await,
        }
    }
}

# 11. 模型侧适配层

FAP 的 wire protocol 是 Protobuf，但模型可以使用更友好的文本语法。

## 11.1 支持两种模型调用格式

### 格式 A：JSON Tool Schema

适合支持结构化输出的模型。

```json id="i9c1dx"
{
  "tool": "code.review",
  "input": {
    "path": "src/main.rs"
  }
}
```

### 格式 B：FAP Text Tool Syntax

适合不稳定或不支持 function calling 的模型。

```text id="z93xd5"
<<<[FAP_TOOL_REQUEST]>>>
capability:「始」code.review「末」
path:「始」src/main.rs「末」
mode:「始」strict「末」
<<<[END_FAP_TOOL_REQUEST]>>>
```

解析后统一转换为：

```text id="zne5tb"
InvokeRequest
```

这样可以吸收 VCP 的“模型友好语法”思想，但不依赖 VCPToolBox。

# 19. 最终版本定义

最终你要做的不是 “FAP + VCPToolBox”，而是：

```text id="m6ylnr"
FAP-1 Core:
  稳定、安全、强类型、可审计的 Agent 通信协议。

FAP-HP Profile:
  吸收 VCP 思想后重新设计的高性能能力层。

FAP Plugin Runtime:
  完整可插拔插件架构，支持 Rust/WASM/Java Sidecar。

FAP Extension Profile:
  潜空间、Gossip、CRDT、DID/VC 等研究能力的受控实验区。
```
