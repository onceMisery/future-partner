# 第 1 章补丁：设计目标增加“轻量化内核”

在 `1.1 设计目标` 中新增：

## 1.1.1 轻量化内核目标

FAP Core 必须可以在不启用 FAP-HP、不启用分布式节点、不启用 WASM 插件、不启用 DID/VC 的情况下独立运行。

最小运行形态：

```text
FAP Core Lite =
  Agent Card
  HTTP JSON Gateway
  Protobuf Envelope
  ClientHello / ServerHello
  mTLS/JWT 或本地 Token
  Capability Registry
  InvokeRequest / InvokeResult
  Receipt
  SQLite WAL
  Idempotency
  Replay Cache
```

目标部署环境：

| 环境       | 要求                   |
| -------- | -------------------- |
| 低功耗设备    | 可运行 Core Lite        |
| 单机开发环境   | 不需要 QUIC、不需要分布式节点    |
| 边缘 Agent | 可关闭 FAP-HP 和 FAP-EXT |
| 企业网关     | 可按需启用 QUIC、插件、分布式节点  |
| 研究集群     | 可启用 FAP-EXT 实验能力     |

设计约束：

```text
FAP Core 不依赖 FAP-HP。
FAP Core 不依赖 WASM。
FAP Core 不依赖 QUIC。
FAP Core 不依赖 DID/VC。
FAP Core 不依赖分布式节点。
FAP Core 必须可以退化为 HTTP JSON + Protobuf + SQLite。
```

## 4.2 Agent Card 版本字段规范

## 4.5 SemVer 兼容规则

| 版本变化       | 兼容性    | 策略                     |
| ---------- | ------ | ---------------------- |
| PATCH      | 必须兼容   | 自动接受                   |
| MINOR      | 向后兼容   | 可自动协商                  |
| MAJOR      | 可能破坏兼容 | 必须显式协商                 |
| PRERELEASE | 不稳定    | 默认拒绝，除非启用 experimental |
| BUILD      | 不影响协议  | 仅用于追踪构建                |

Agent Card 校验规则：

```text
client.protocol.major == server.protocol.major
client.protocol.version >= server.minCompatibleVersion
required profile 必须满足 major 相同
optional profile 可降级关闭
plugin_api major 必须一致
```

# 第 7 章补丁：Data Plane 增加动态背压

在 `第 7 章：数据面设计` 中新增：

## 7.5 动态背压设计

Data Plane 必须实现基于 QUIC 流控和 FAP 应用层信号的双层背压机制。

目标：

```text
防止大文件传输拖垮 Agent 逻辑线程；
防止 Data Plane 挤占 Control Plane；
防止慢节点造成内存无限堆积；
防止远程对象流在网络抖动下形成雪崩重试。
```

### 7.5.1 背压层级

| 层级             | 机制                                    | 作用       |
| -------------- | ------------------------------------- | -------- |
| QUIC Transport | stream window / connection window     | 控制底层传输速率 |
| FAP Data Plane | FlowCredit / DataAck / Pause / Resume | 控制应用层吞吐  |
| Scheduler      | per-session quota / per-node quota    | 防止任务饥饿   |
| Memory Guard   | max buffered bytes                    | 防止内存被打满  |
| Priority Queue | control > audit > data                | 保证控制面优先  |

### 7.5.2 新增 Protobuf

```proto
message FlowCredit {
  string session_id = 1;
  string stream_id = 2;

  uint64 granted_bytes = 3;
  uint32 granted_chunks = 4;

  uint64 valid_until_sequence_no = 5;
  uint32 recommended_chunk_bytes = 6;

  FlowPressure pressure = 7;
}

enum FlowPressure {
  FLOW_PRESSURE_UNSPECIFIED = 0;
  FLOW_PRESSURE_NORMAL = 1;
  FLOW_PRESSURE_WARN = 2;
  FLOW_PRESSURE_SLOW_DOWN = 3;
  FLOW_PRESSURE_PAUSE = 4;
  FLOW_PRESSURE_CANCEL = 5;
}

message DataPause {
  string stream_id = 1;
  string reason = 2;
  uint32 retry_after_ms = 3;
}

message DataResume {
  string stream_id = 1;
  uint64 next_offset = 2;
  uint32 recommended_chunk_bytes = 3;
}
```

Envelope 增加：

```proto
oneof body {
  FlowCredit flow_credit = 120;
  DataPause data_pause = 121;
  DataResume data_resume = 122;
}
```

### 7.5.3 背压算法

接收端维护：

```text
available_memory_bytes
session_buffered_bytes
stream_buffered_bytes
control_queue_depth
data_queue_depth
disk_flush_latency_ms
```

压力评分：

```text
pressure_score =
  0.35 * memory_pressure
+ 0.25 * queue_pressure
+ 0.20 * disk_pressure
+ 0.10 * cpu_pressure
+ 0.10 * network_retransmit_pressure
```

策略：

| pressure_score | 动作                      |
| -------------: | ----------------------- |
|         < 0.50 | NORMAL                  |
|      0.50～0.70 | WARN，降低 chunk size      |
|      0.70～0.85 | SLOW_DOWN，减少 FlowCredit |
|      0.85～0.95 | PAUSE                   |
|         > 0.95 | CANCEL 低优先级 data stream |

### 7.5.4 控制面优先级

必须保证：

```text
Control Stream 永远优先于 Data Stream。
Receipt / Problem / Cancel 优先于普通 ProgressEvent。
Data Plane 不得阻塞 Auth、Cancel、Receipt。
```

# 第 13 章替换：Receipt 多签审计模型

原文 Receipt 已经定义了 request_hash、response_hash、transcript_hash、previous_receipt_hash 与 signatures 字段。建议正式升级为 Multisig Receipt。

## 13.1 Receipt 目标

Receipt 不只是调用日志，而是可验证审计凭证。它必须证明：

```text
请求确实由某主体发起；
能力确实由某执行节点执行；
输出确实对应该输入；
高风险动作确实基于有效 Mandate；
审计中心确实接收并确认；
Receipt 未被篡改；
Receipt 链未断裂。
```

## 13.2 多签角色

| 签名角色           | 必选          | 说明              |
| -------------- | ----------- | --------------- |
| REQUESTER      | 可选          | 请求方签名，证明请求意图    |
| EXECUTOR_NODE  | 必选          | 执行节点签名，证明执行结果   |
| MAIN_NODE      | 必选          | 主节点签名，证明路由与策略校验 |
| AUDIT_CENTER   | 高风险必选       | 审计中心签名          |
| HUMAN_APPROVER | critical 必选 | 人工审批签名或审批引用     |
| STORAGE_NODE   | 涉及远程对象时必选   | 证明对象 hash 与可用性  |

## 13.3 Protobuf 替换

```proto
message Receipt {
  string receipt_id = 1;
  string invocation_id = 2;
  string batch_id = 3;

  string subject = 4;
  string capability_id = 5;
  string mandate_id = 6;

  string executor_node_id = 7;
  string main_node_id = 8;

  string status = 9;

  bytes request_hash = 10;
  bytes response_hash = 11;
  bytes transcript_hash = 12;
  bytes previous_receipt_hash = 13;

  repeated ObjectReceipt object_receipts = 14;

  google.protobuf.Timestamp created_at = 15;

  MultisigPolicy multisig_policy = 20;
  repeated ReceiptSignature signatures = 21;
}

message ObjectReceipt {
  string object_id = 1;
  bytes sha256 = 2;
  uint64 size_bytes = 3;
  string origin_node_id = 4;
  repeated string replica_node_ids = 5;
}

message MultisigPolicy {
  string policy_id = 1;
  uint32 threshold = 2;
  repeated SignerRequirement required_signers = 3;
}

message SignerRequirement {
  SignerRole role = 1;
  bool required = 2;
}

enum SignerRole {
  SIGNER_ROLE_UNSPECIFIED = 0;
  SIGNER_REQUESTER = 1;
  SIGNER_EXECUTOR_NODE = 2;
  SIGNER_MAIN_NODE = 3;
  SIGNER_AUDIT_CENTER = 4;
  SIGNER_HUMAN_APPROVER = 5;
  SIGNER_STORAGE_NODE = 6;
}

message ReceiptSignature {
  SignerRole role = 1;
  string signer_id = 2;
  string alg = 3;
  string kid = 4;
  bytes signature = 5;
  google.protobuf.Timestamp signed_at = 6;
}
```

## 13.4 多签策略

| 风险等级     | 策略                                                        |
| -------- | --------------------------------------------------------- |
| low      | MAIN_NODE 单签即可                                            |
| medium   | MAIN_NODE + EXECUTOR_NODE                                 |
| high     | MAIN_NODE + EXECUTOR_NODE + AUDIT_CENTER                  |
| critical | MAIN_NODE + EXECUTOR_NODE + AUDIT_CENTER + HUMAN_APPROVER |

## 13.5 Receipt 验证流程

```text
1. 校验 receipt_id 格式。
2. 校验 request_hash。
3. 校验 response_hash。
4. 校验 transcript_hash。
5. 校验 previous_receipt_hash。
6. 读取 multisig_policy。
7. 检查 required_signers 是否齐全。
8. 校验每个签名的 kid 和证书链。
9. 校验签名时间在会话有效期内。
10. 校验高风险动作是否有 AUDIT_CENTER 签名。
11. 校验 critical 动作是否有人类审批引用。
```

# 第 12 章补丁：DID/VC 性能优化

在 `12.1 默认安全模型` 后新增：

## 12.5 DID/VC 性能保护

DID/VC 仅作为 FAP-EXT 的跨组织认证插件，默认关闭。启用后必须避免每次握手都同步访问外部证书库或撤销列表。

### 12.5.1 缓存策略

| 数据                           |    缓存 TTL | 说明                        |
| ---------------------------- | --------: | ------------------------- |
| DID Document                 |   5～30 分钟 | 按 DID method 配置           |
| VC Schema                    |   1～24 小时 | Schema 变化低频               |
| Issuer Public Key            |   5～30 分钟 | 需支持 key rotation          |
| Revocation Status            | 30 秒～5 分钟 | 高风险场景 TTL 更短              |
| Verified Presentation Result |     会话级缓存 | 绑定 session_id 和 challenge |

### 12.5.2 异步预取

```text
Agent Card 发现阶段：
  -> 解析 did method
  -> 预取 DID Document
  -> 预取 issuer metadata
  -> 预热 revocation endpoint

ClientHello 阶段：
  -> 生成 challenge
  -> 使用缓存验证 DID Document
  -> 并行检查撤销状态

AuthAck 阶段：
  -> 返回 auth_context_id
  -> 后续 Invoke 复用 auth_context_id
```

### 12.5.3 性能降级策略

| 情况                     | 策略                           |
| ---------------------- | ---------------------------- |
| DID resolver 超时        | 返回 AUTH_PENDING 或降级拒绝        |
| Revocation endpoint 超时 | high/critical 必须 fail-closed |
| medium/low 风险          | 可配置短期 soft-fail              |
| 缓存过期                   | 后台刷新，不阻塞已有会话                 |
| key rotation           | 触发 auth_context 失效           |

新增配置：

```yaml
did_vc:
  enabled: false
  did_document_cache_ttl_seconds: 900
  revocation_cache_ttl_seconds: 60
  resolver_timeout_ms: 800
  auth_context_ttl_seconds: 1800
  high_risk_fail_closed: true
  async_prefetch: true
```

# 第 20 章补丁：测试设计增强

新增必测场景：

```text
SemVer：
  Agent Card 非 SemVer 版本被拒绝
  MAJOR 不一致时拒绝
  MINOR 向后兼容时允许
  PRERELEASE 默认拒绝
  plugin_api major 不一致时插件不加载

Receipt Multisig：
  缺少 executor 签名被拒绝
  high 风险缺少 audit center 签名被拒绝
  critical 缺少 human approval 被拒绝
  signature kid 不可信被拒绝
  previous_receipt_hash 断链被拒绝

Data Backpressure：
  接收端内存压力升高时发送 SLOW_DOWN
  接收端磁盘阻塞时发送 PAUSE
  控制面消息不被 DataChunk 阻塞
  CANCEL 能中断大文件流
  chunk size 能根据 FlowCredit 动态调整

ObjectRef：
  TTL 过期对象不可调用
  origin_node 掉线后切换 replica
  所有副本失效时返回 OBJECT_UNAVAILABLE
  GC 不删除仍有 lease 的对象

DID/VC：
  resolver 超时不阻塞低风险已有会话
  high 风险撤销检查失败时 fail-closed
  auth_context 绑定 session_id
  VP 重放到其他 session 被拒绝

Core Lite：
  禁用 QUIC 后 HTTP JSON 可运行
  禁用 WASM 后内置 capability 可运行
  禁用 FAP-HP 后基本 Invoke 可运行
```

# 新增第 26 章：风险清单与缓解策略

## 26.1 复杂度陷阱

风险：

```text
分布式能力节点 + QUIC + WASM 隔离 + DID/VC 同时推进，导致 MVP 过重。
```

缓解：

```text
Core Lite 先行。
QUIC 后置。
WASM 后置。
分布式节点后置。
DID/VC 后置。
FAP-Inspector 前置。
```

## 26.2 存储一致性风险

风险：

```text
ObjectRef 指向远程对象，网络分区后对象不可用。
```

缓解：

```text
ObjectRef 必须有 TTL。
远程对象必须有 availability check。
重要对象必须 replica。
调用期间必须持有 lease。
过期对象进入 GC。
```

## 26.3 DID/VC 性能瓶颈

风险：

```text
频繁访问 DID resolver、VC schema、revocation list，拉高握手延迟。
```

缓解：

```text
DID Document 缓存。
Revocation 状态短 TTL 缓存。
AuthContext 会话级复用。
异步预取。
高风险 fail-closed，低风险可配置 soft-fail。
```

## 26.4 插件兼容风险

风险：

```text
插件升级导致 schema 不兼容，运行时崩溃。
```

缓解：

```text
plugin_api 使用 SemVer。
MAJOR 不一致拒绝加载。
MINOR 兼容自动加载。
PATCH 自动加载。
插件必须声明 min_core_version。
插件加载前执行 schema dry-run。
```

## 26.5 Data Plane 拖垮控制面风险

风险：

```text
大文件传输占满连接、内存或磁盘，导致 Cancel、Problem、Receipt 无法及时处理。
```

缓解：

```text
Control Stream 最高优先级。
FlowCredit 动态背压。
DataPause / DataResume。
低优先级流可取消。
每 session 限制 buffer。
```

# 修改后的最终结论

改进后的 FAP-1 文档应从：

```text
FAP Core + FAP-HP + FAP-EXT
```

升级为：

```text
FAP Core Lite:
  可在低功耗设备运行的最小协议内核。

FAP Core Hardening:
  SemVer、Receipt Multisig、Replay、Idempotency、Hash Chain。

FAP Data Stability:
  Dynamic Backpressure、ObjectRef TTL、Lease、Replica、GC。

FAP Developer Experience:
  FAP-Inspector、Golden Replay、Envelope Decoder、Receipt Verifier。

FAP-HP:
  动态工具注入、上下文折叠、批量并行、分布式能力节点。

FAP-EXT:
  DID/VC、潜空间、Gossip、CRDT，全部默认关闭。
```

核心改进方向是：**先让 FAP Core 轻、稳、可调试，再逐步叠加高性能与研究能力。**
