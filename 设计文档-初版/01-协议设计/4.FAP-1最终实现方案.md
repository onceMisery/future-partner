## 1. 方案定位

FAP-1（Future Agent Protocol 1）是一个面向 Agent 生态的原生通信协议。

FAP-1 不是：MCP 复制品、A2A 复制品、VCPToolBox 移植版、单一工具调用格式、某个插件框架。

FAP-1 是：发现 + 会话 + 能力 + 安全授权 + 数据传输 + 审计 + 插件运行时 + 分布式节点协作的完整栈。

### 1.1 超越 VCPToolBox 的关键差异

| 维度 | VCP | FAP-1 |
|---|---|---|
| 传输 | WebSocket + HTTP | QUIC/HTTP3/HTTP JSON 三栈 |
| 文件传输 | Base64 回源 | ObjectRef + DataPlane + Lease |
| 审计 | 弱/无 | Multisig Receipt + Hash Chain + Transparency Log |
| 插件运行时 | Node.js 单一 | Rust Native + WASM + Sidecar 三运行时 |
| 安全 | API Key 为主 | mTLS/JWT + Mandate 委托链 + DID/VC 可选 |
| 批量执行 | 异步并行 | BatchInvoke + DAG depends_on |
| 分布式 | WebSocket Worker | QUIC Node Network + Health FSM + 断路器 |
| 版本治理 | 无明确策略 | SemVer + Wire Version + 降级防护 |
| 研究能力 | 混在核心 | EXT Profile 默认关闭，编译期可剔除 |

### 1.2 三层协议模型

```text
FAP Core:
  稳定通信、安全、会话、能力、数据、审计。
  必须，最小形态为 Core Lite。

FAP-HP Profile:
  动态工具注入、上下文折叠、DAG 批量调用、分布式节点、统一文件解析。
  推荐，按需启用。

FAP-EXT Profile:
  DID/VC、潜空间通信、Gossip 发现、CRDT 工作区、联邦学习。
  实验，默认关闭，可编译剔除。
```

## 2. 设计目标与非目标

### 2.1 核心目标

| 目标 | 说明 |
|---|---|
| 独立协议 | 不依赖 MCP/A2A/VCP，可桥接但不绑定 |
| 轻量化内核 | Core Lite 可在单机/边缘/低功耗设备运行 |
| 高性能 | QUIC 多路复用、DAG 批量、零拷贝数据面 |
| 可插拔 | 能力/传输/认证/检索/桥接全部插件化 |
| 强安全 | mTLS + JWT + Mandate 委托链 + Replay 防护 |
| 可审计 | 关键调用生成可验证 Multisig Receipt |
| 可演进 | SemVer + 降级防护 + Schema breaking CI |
| 可观测 | OpenTelemetry + FAP-Inspector 工具链前置 |

### 2.2 非目标

```text
不默认启用潜空间通信
不默认启用 Gossip 网络
不默认启用 CRDT 协同
不默认启用全量 DID/VC
不强制所有模型使用 JSON Function Calling
不复制 VCPToolBox 实现结构
```

### 2.3 轻量内核约束

```text
FAP Core Lite 不依赖 FAP-HP
FAP Core Lite 不依赖 WASM
FAP Core Lite 不依赖 QUIC
FAP Core Lite 不依赖 DID/VC
FAP Core Lite 不依赖分布式节点
FAP Core Lite 必须可退化为 HTTP JSON + Protobuf + SQLite
```

## 4. 协议平面划分

原方案 L0–L9 十层过度抽象。FAP-1 收敛为 **4 平面**：

| 平面 | 职责 | 载体 |
|---|---|---|
| Discovery Plane | Agent Card、能力目录 | HTTPS `/.well-known/agent-card.json` |
| Session Plane | Hello、版本协商、Auth、Resume | Envelope over QUIC/HTTP |
| Invocation Plane | Control + Data + Audit 三股流共享通道 | Envelope 元数据 + raw data stream |
| Extension Plane | EXT 特性，编译期可剔除 | Envelope extension field |

Invocation Plane 内部按消息类型分优先级队列：

```text
Priority 0: Control (Hello/Auth/Capability/Cancel/Problem)
Priority 1: Audit (Receipt)
Priority 2: Data (DataOpen/DataChunkMeta/DataCommit)
```

控制面消息必须优先于数据面消息出队。

### 5.3 SemVer 规则

| 变化 | 兼容性 | 策略 |
|---|---|---|
| PATCH | 必须兼容 | 自动接受 |
| MINOR | 向后兼容 | 可自动协商 |
| MAJOR | 可能破坏 | 必须显式协商 |
| PRERELEASE | 不稳定 | 默认拒绝，仅 allowlist 精确匹配时允许 |
| BUILD | 不影响协议 | 仅追踪构建 |

`acceptPrereleases` 两档：

| 值 | 行为 |
|---|---|
| none | 拒绝任何 prerelease |
| allowlist | 仅接受 `prereleaseAllowlist` 中精确列出的版本 |

### 5.4 Agent Card 校验

```text
1. 取回 agent-card.json
2. 如果 cardSignature 存在，校验签名（kid 指向已信任的 issuer）
3. protocol.major == client.protocol.major
4. protocol.version >= client.minCompatibleVersion
5. required profile 必须都能满足
6. optional profile 可降级关闭
7. pluginApi.major 必须一致
```

## 6. 会话层

### 6.1 状态机

```text
NEW
  → DISCOVERED
  → HELLO_EXCHANGED
  → AUTHENTICATED
  → CAPABILITY_BOUND
  → ACTIVE
  → DRAINING
  → CLOSED
```

异常状态：AUTH_FAILED / POLICY_REJECTED / EXPIRED / RESUMING / BROKEN。

### 6.2 版本协商流程

```text
ClientHello:
  client_protocol_version
  supported_protocol_range (min..max)
  supported_profiles
  supported_plugin_api_versions
  client_nonce
  supported_transports
  supported_auth_modes

ServerHello:
  selected_protocol_version
  selected_profiles
  rejected_profiles
  selected_plugin_api_version
  compatibility_warnings
  session_id
  server_nonce
  handshake_transcript_hash
  version_commitment         ← 防降级
```

### 6.3 防降级攻击

服务端必须生成：

```text
version_commitment =
  HMAC(session_key,
       SHA256(client_hello.supported_protocol_range
              || client_hello.supported_profiles
              || client_hello.client_nonce))
```

客户端收到 ServerHello 后本地重算并比对；不一致则拒绝会话。

### 6.4 会话恢复

```text
客户端断线
  → 重连
  → ResumeRequest(session_id, last_received_sequence_no)
  → 服务端校验会话有效 + 鉴权凭证未过期
  → ResumeAck(replay_from_sequence_no)
  → 重放未确认 ProgressEvent / Result / Receipt
```

### 6.5 QUIC 0-RTT 策略

```text
默认：禁用 0-RTT
Resume 场景：允许 0-RTT，但 Early Data 只接受幂等消息
            (InvokeRequest 必须带 idempotency_key，
             Replay Cache 命中时必须返回缓存结果而非拒绝)
high/critical 风险：强制 full handshake，0-RTT 拒绝
```

```proto
  string protocol_version = 1;          // SemVer
  string min_protocol_version = 2;      // SemVer

  uint32 wire_major = 4;                // = protocol MAJOR
  uint32 wire_minor = 5;                // = protocol MINOR

  string message_id = 6;

  google.protobuf.Timestamp expires_at = 14;

  string trace_id = 15;
  string parent_message_id = 16;

  bytes prev_hash = 18;

```

说明：`schema_version` 已删除，与 `protocol_version` 合并；schema 与协议 lockstep 演进。

### 7.3 控制面校验顺序

```text
1. Decode Protobuf
2. 校验 wire_major/wire_minor
3. 校验 message_id 格式
4. 校验 expires_at ≤ session.max_ttl
5. 校验 Replay Cache
6. 校验 session_id
7. 校验 payload_sha256
8. 校验 prev_hash 指向链内
9. 校验 signatures（按签名规范化规则）
10. 分发 handler
```

### 7.4 签名规范化

签名字节定义：

```text
signed_bytes = protobuf_deterministic_encode(EnvelopeSigningView)

EnvelopeSigningView = Envelope {
  header,
  body
  // signatures 字段必须从 signing view 排除
}

规范化要求：
- proto3 canonical serialization
- map<string,string> 按 key 字典序排序
- 嵌套 Struct 按 proto binary（非 JSON）
- 所有 Timestamp 以 nanos 精度编码
```

配套交付：`docs/fap/signing-canonical.md` 含字节级测试向量。

## 8. 数据面

### 8.1 ObjectRef

```proto

  string object_id = 1;
  string media_type = 2;
  uint64 size_bytes = 3;
  bytes sha256 = 4;

  string origin_node_id = 5;

  google.protobuf.Timestamp created_at = 7;
  google.protobuf.Timestamp expires_at = 8;
  uint32 ttl_seconds = 9;

  ObjectAvailability availability = 10;
  repeated string replica_node_ids = 11;
  string lease_id = 12;
}

enum ObjectAvailability {
  OBJECT_AVAILABILITY_UNSPECIFIED = 0;
  OBJECT_LOCAL = 1;
  OBJECT_REMOTE_AVAILABLE = 2;
  OBJECT_REPLICATED = 3;
  OBJECT_EXPIRED = 4;
  OBJECT_UNAVAILABLE = 5;
}
```

### 8.2 生命周期

```text
CREATED → LOCAL / REMOTE_AVAILABLE / REPLICATED
       → EXPIRED → GC_PENDING → DELETED
```

规则：

```text
ObjectRef 必须有 created_at
临时对象必须有 expires_at 或 ttl_seconds
跨节点 ObjectRef 必须有 origin_node_id
远程对象必须支持 availability check
高价值对象必须复制到 ≥ 1 个 replica
过期对象不可被新 InvokeRequest 引用
```

### 8.3 Lease 续租

```proto
message ObjectLeaseRenew {
  string object_id = 1;
  string lease_id = 2;
  uint32 requested_extension_seconds = 3;
}

message ObjectLeaseRenewAck {
  string lease_id = 1;
  google.protobuf.Timestamp new_expires_at = 2;
  bool granted = 3;
  string reject_reason = 4;
}
```

规则：

```text
Lease 持有期间 object 不可被 GC
续租上限 = max(3 × 原 TTL, 3600s)，超过拒绝
续租需由原 lease 持有者签名发起
```

### 8.5 动态背压

双层机制，互不冲突：

| 层 | 机制 | 作用 |
|---|---|---|
| QUIC Transport | stream window / connection window | 字节级调度 |
| FAP Data Plane | FlowCredit / DataPause / DataResume | 语义级背压 |

```proto
message FlowCredit {
  string session_id = 1;
  string stream_id = 2;

  uint64 granted_bytes = 3;
  uint32 granted_chunks = 4;

  uint64 valid_until_sequence_no = 5;
  uint32 recommended_chunk_bytes = 6;

  FlowPressure pressure = 7;
  FlowTransportHint transport_hint = 8;
}

enum FlowPressure {
  FLOW_PRESSURE_UNSPECIFIED = 0;
  FLOW_PRESSURE_NORMAL = 1;
  FLOW_PRESSURE_WARN = 2;
  FLOW_PRESSURE_SLOW_DOWN = 3;
  FLOW_PRESSURE_PAUSE = 4;
  FLOW_PRESSURE_CANCEL = 5;
}

enum FlowTransportHint {
  FLOW_TRANSPORT_HINT_UNSPECIFIED = 0;
  QUIC_PIGGYBACK = 1;   // 语义级，与 QUIC 流控叠加
  HTTP_PRIMARY = 2;     // HTTP 下作为主要背压
}

message DataPause {
  string stream_id = 1;
  string reason = 2;
  uint32 retry_after_ms = 3;
}

message DataResume {
  string stream_id = 1;
  uint64 next_offset = 2;
  uint32 recommended_chunk_bytes = 3;
}
```

协同规则：

```text
QUIC 模式下：
  FlowCredit.granted_bytes 必须 ≥ QUIC MAX_STREAM_DATA 剩余，否则无效
  FlowCredit 只用于语义信号，不参与字节调度

HTTP 模式下：
  FlowCredit 作为主要背压手段
  SSE 心跳承载 pressure 状态
```

### 8.6 事件驱动 Pressure

不使用加权公式，改用独立阈值触发：

```text
MEM:   buffered_bytes > memory_high_water   → SLOW_DOWN
       buffered_bytes > memory_pause_water  → PAUSE
DISK:  fsync_p99_ms > 200                   → SLOW_DOWN
QUEUE: data_queue_depth > cap * 0.8         → SLOW_DOWN
NET:   retransmit_rate > 5%                 → SLOW_DOWN
CPU:   load_avg > cores * 0.9               → WARN

恢复: 全部维度 < low_water 持续 10s → NORMAL
```

每个维度阈值在配置中单独可调，并由 OpenTelemetry 暴露实时指标。

### 8.7 远程文件解析

```text
插件引用 file://node-a/path/to/a.png
  → RemoteFileResolver 判断本地不存在
  → 定位 origin_node_id
  → 向 node-a 发起 DataOpen
  → 通过 raw stream 流式拉取分片（Envelope 仅承载 DataChunkMeta）
  → DataCommit 校验 hash
  → 生成 ObjectRef（含 lease_id）
  → 替换插件输入
  → 调用插件
  → 结束后释放 lease
```

### 8.8 网络分区处理

| 场景 | 行为 |
|---|---|
| origin_node 不可达 | 标记 OBJECT_UNAVAILABLE |
| replica 可用 | 切换 replica |
| 所有副本不可用 | 返回 Problem: OBJECT_UNAVAILABLE |
| ObjectRef 已过期 | 拒绝新调用 |
| 调用执行中对象过期 | 先尝试续租，失败则 Problem: LEASE_EXPIRED |
| 节点恢复 | 重新执行 availability check |

```json

  "issuer": "did:web:alice.example",
```

```text

```

### 11.4 撤销与 AuthRevoke

```proto
message AuthRevoke {
  string session_id = 1;
  string mandate_id = 2;
  string reason = 3;
  uint32 retry_after_ms = 4;
  RevokeScope scope = 5;
}

  REVOKE_SCOPE_UNSPECIFIED = 0;
  REVOKE_MANDATE = 1;
  REVOKE_SESSION = 2;
  REVOKE_SUBJECT = 3;
}
```

策略：

```text
每次 InvokeRequest 查询撤销表（缓存 TTL ≤ 60s）
每 5 分钟做一次 session-wide revocation sync
发现撤销 → session 转 DRAINING，在途 Invoke 完成，不接受新 Invoke
high/critical 能力必须每次 re-check
low/medium 能力可会话级缓存
```

### 11.5 Replay 防护

```text
校验维度：
  message_id 未使用
  idempotency_key 未冲突
  created_at 在时钟偏移窗内
  expires_at 未过期且 ≤ session.max_ttl
  payload_sha256 匹配
  prev_hash 指向链内
  session_id 有效
  nonce 绑定当前 session

Cache 实现：
  time-windowed bloom + exact cache
  精确缓存保留 max(expires_at) + 5 min 内所有 message_id
  多节点部署必须共享（Redis/etcd）
  未共享时 idempotency 粒度限制到 node-local
  拒绝 expires_at 缺失或 > 15 min 的消息
```

### 11.7 DID/VC 性能保护（EXT Profile）

| 数据 | 缓存 TTL | 说明 |
|---|---|---|
| DID Document | 5~30 分钟 | 按 DID method |
| VC Schema | 1~24 小时 | 变化低频 |
| Issuer Public Key | 5~30 分钟 | 支持 key rotation |
| Revocation Status | 30 秒~5 分钟 | 高风险更短 |

| 情况 | 策略 |
|---|---|

## 12. 审计模型

### 12.1 Receipt 目标

Receipt 不是日志，是可验证审计凭证，必须证明：

```text
请求确实由某主体发起
能力确实由某执行节点执行
输出确实对应该输入
高风险动作确实基于有效 Mandate
审计中心确实接收并确认
Receipt 未被篡改
Receipt 链未断裂
```

### 12.2 Multisig 角色

| 签名角色 | 必选 | 说明 |
|---|---|---|
| REQUESTER | 可选 | 请求方签名，证明请求意图 |
| EXECUTOR_NODE | 必选 | 执行节点签名 |
| MAIN_NODE | 必选 | 主节点签名 |
| AUDIT_CENTER | high 必选 | 审计中心签名 |
| HUMAN_APPROVER | critical 必选 | 人工审批签名或审批引用 |
| STORAGE_NODE | 涉远程对象时必选 | 对象 hash 与可用性 |

### 12.3 Receipt 定义

```proto
message Receipt {
  string receipt_id = 1;
  string invocation_id = 2;
  string batch_id = 3;

  string capability_id = 5;

}

  RECEIPT_FINALITY_UNSPECIFIED = 0;
  RECEIPT_PENDING = 1;         // 多签未齐
  RECEIPT_PROVISIONAL = 2;     // 关键签名已齐，异步补签中

}

  string object_id = 1;

  uint64 size_bytes = 3;

}

  string policy_id = 1;
  uint32 threshold = 2;
  repeated SignerRequirement required_signers = 3;
}

message SignerRequirement {
  SignerRole role = 1;
  bool required = 2;
}

enum SignerRole {
  SIGNER_ROLE_UNSPECIFIED = 0;
  SIGNER_REQUESTER = 1;
  SIGNER_EXECUTOR_NODE = 2;
  SIGNER_MAIN_NODE = 3;
  SIGNER_AUDIT_CENTER = 4;

  SIGNER_STORAGE_NODE = 6;
```

### 12.4 风险 → 签名策略

| 风险 | 必选签名 |
|---|---|
| low | MAIN_NODE |
| medium | MAIN_NODE + EXECUTOR_NODE |
| high | MAIN_NODE + EXECUTOR_NODE + AUDIT_CENTER |
| critical | MAIN_NODE + EXECUTOR_NODE + AUDIT_CENTER + HUMAN_APPROVER |

### 12.5 异步补签

降低 RTT 影响的关键：

```text
调用返回时：
  EXECUTOR + MAIN 在线签 → Receipt 状态 = PROVISIONAL 立即返回给客户端

后台：
  AUDIT_CENTER 异步签 → 状态变 PROVISIONAL（若已是）
  HUMAN_APPROVER 异步签或引用预授权 mandate → 状态变 FINALIZED
  推送 ReceiptFinalized 事件给订阅方

约束：
  PROVISIONAL Receipt 可用于返回客户端，但不得用于对账/结算
  超过 24h 未 FINALIZED 的 Receipt 触发告警
  critical 能力若无法获得 HUMAN_APPROVER，必须回滚并生成补偿 Receipt
```

### 12.6 Hash Chain 锚定

```text
会话首 Receipt 的 previous_receipt_hash =
  SHA256(session_id || server_nonce || handshake_transcript_hash)

跨会话不强制延续，但同 subject 的 Receipt 链可可选聚合

节点重启：
  从 SQLite session.hash_chain_tip 恢复
  启动前做 chain 完整性校验（receipt 表自洽）
  发现断链 → 进入 DEGRADED 模式，拒绝 high/critical 请求直到人工介入

Checkpoint 机制（Transparency Log）：
  每日或每 N=10000 条 Receipt 生成一个 checkpoint
  checkpoint = Merkle root of 该周期所有 receipt_hash
  推送到外部 transparency log（默认本地 append-only log）
  支持 Sigstore Rekor 等外部 log 配置
```

### 12.7 Receipt 验证流程

```text
1. 校验 receipt_id 格式
2. 校验 request_hash 与 invocation 记录一致
3. 校验 response_hash 与 invocation 记录一致
4. 校验 transcript_hash
5. 校验 previous_receipt_hash 指向链内
6. 读取 multisig_policy
7. 检查 required_signers 齐全（按 finality 状态判断）
8. 校验每个签名的 kid 和证书链
9. 校验签名时间在会话有效期内
10. 校验 high 动作有 AUDIT_CENTER 签名
11. 校验 critical 动作有 HUMAN_APPROVER 签名或 mandate 引用
12. 校验 object_receipts 与实际 ObjectRef 一致
```

### 14.4 调度策略

```text
短任务:      local cold start
高频任务:    warm worker pool
长任务:      async task (AsyncTask 消息订阅结果)
远程重任务:  remote node
批量 DAG:    parallel scheduler + 拓扑排序
```

Rust 调度入口：

```rust
pub struct ExecutionScheduler {
    local_pool: LocalWorkerPool,
    warm_pool: WarmWorkerPool,
    remote_router: RemoteCapabilityRouter,
    idempotency: IdempotencyStore,
    receipt: ReceiptService,
    node_health: NodeHealthTable,
}

impl ExecutionScheduler {
    pub async fn execute(&self, req: InvokeRequest) -> Result<InvocationResult, FapError> {
        let plan = self.plan(&req).await?;

        match plan.mode {
            ExecutionMode::LocalColdStart => self.local_pool.spawn_once(req).await,
            ExecutionMode::LocalWarmWorker => self.warm_pool.invoke(req).await,
            ExecutionMode::RemoteNode(node_id) => {
                self.node_health.check_healthy(&node_id)?;
                self.remote_router.forward(node_id, req).await
            }
            ExecutionMode::AsyncTask => self.spawn_async(req).await,
        }
    }
}
```

## 17. HTTP/JSON Fallback 绑定

Core Lite 必须支持 HTTP JSON 降级。

### 17.1 Endpoint 映射

```text
POST /fap/v1/session/hello          body = ClientHello JSON → ServerHello JSON
POST /fap/v1/session/auth           body = AuthInit JSON → AuthAck JSON
POST /fap/v1/capability/bind        body = CapabilityBind JSON → CapabilityAck JSON
POST /fap/v1/invoke                 body = InvokeRequest JSON → InvokeResult JSON
GET  /fap/v1/invoke/:id/stream      SSE 流接收 ProgressEvent
POST /fap/v1/batch                  body = BatchInvokeRequest JSON
POST /fap/v1/data/upload            multipart，承载 raw chunks，Envelope 只签 DataOpen/DataCommit
GET  /fap/v1/data/:object_id        流式下载 ObjectRef
POST /fap/v1/resume                 body = ResumeRequest JSON
GET  /fap/v1/receipts/:id           返回 Receipt JSON
```

### 17.2 Header 映射

Envelope Header 字段映射到 HTTP Header：

```text
X-FAP-Protocol-Version      ← protocol_version
X-FAP-Message-Id            ← message_id
X-FAP-Session-Id            ← session_id
X-FAP-Sequence-No           ← sequence_no
X-FAP-Idempotency-Key       ← idempotency_key
X-FAP-Source-Agent          ← source_agent_id
X-FAP-Target-Agent          ← target_agent_id
X-FAP-Expires-At            ← expires_at (ISO8601)
X-FAP-Trace-Id              ← trace_id
X-FAP-Payload-Sha256        ← hex(payload_sha256)
X-FAP-Prev-Hash             ← hex(prev_hash)
X-FAP-Priority              ← priority
X-FAP-Signature             ← 签名（CSV：alg,kid,value）
```

### 17.3 JSON 编码

使用 proto3 canonical JSON（`google.protobuf.util.JsonFormat`）；所有签名的规范化仍基于 protobuf deterministic encoding，而非 JSON。即：

```text
HTTP JSON 只是线格式，签名计算必须先把 JSON 还原为 Envelope proto
再按 7.4 节规范化规则重新编码得到 signed_bytes。
```

### 17.4 背压降级

HTTP JSON 下：

```text
FlowCredit.transport_hint = HTTP_PRIMARY
SSE 心跳 event 携带 pressure 状态
客户端收到 SLOW_DOWN/PAUSE 后应调整请求速率
```

| Data Plane 拖垮控制面 | 高 | Control Stream 最高优先级 + FlowCredit |
| 复杂度陷阱（QUIC + WASM + DID 并进） | 高 | Core Lite 先行，其余后置 |
| 插件 API SemVer 未涵盖 host 侧 | 中 | plugin_api 独立 SemVer，双向纳管 |
| Quinn/QUIC 库跨语言互操作不成熟 | 中 | Phase 4 再启用，HTTP JSON 保底 |
| Java SDK 滞后 Rust Core | 中 | 每 Phase 内同步交付，冻结前 smoke test |
| 审计中心单点 | 中 | AuditSink 多实例 + checkpoint 冗余 |
| ObjectRef 网络分区对象不可用 | 中 | TTL + availability check + replica + lease |
| DID/VC 性能瓶颈 | 中 | 缓存 TTL + 异步预取 + 高风险 fail-closed |
| Replay Cache 被大流量 evict 后重放 | 中 | 限制 expires_at 上限 + bloom + 共享 cache |
| Mandate 委托链过深 | 低 | max_delegation_depth + 拒绝无签名委托 |
| EXT 特性被私下开启污染内核 | 低 | 编译期剔除 + runtime_features 固化 |
| Schema 演进破坏兼容 | 低 | buf breaking + deprecated 窗口 + MAJOR 显式协商 |

### B. 关键 Rust 接口

```rust
// 核心分发入口
pub async fn handle_envelope(
    ctx: RequestContext,
    envelope: Envelope,
    services: &Services,
) -> Result<Envelope, FapError> {
    services.validator.validate_header(&envelope.header)?;
    services.replay.check_and_store(&envelope.header)?;
    services.security.verify(&ctx, &envelope).await?;
    services.policy.check(&ctx, &envelope).await?;

    match envelope.body {
        Some(Body::ClientHello(msg))        => services.session.handle_client_hello(ctx, msg).await,
        Some(Body::AuthInit(msg))           => services.security.handle_auth_init(ctx, msg).await,
        Some(Body::CapabilityBind(msg))     => services.capability.handle_bind(ctx, msg).await,
        Some(Body::InvokeRequest(msg))      => services.invocation.invoke(ctx, envelope.header, msg).await,
        Some(Body::BatchInvokeRequest(msg)) => services.invocation.batch_invoke(ctx, envelope.header, msg).await,
        Some(Body::InvokeCancel(msg))       => services.invocation.cancel(ctx, msg).await,
        Some(Body::DataOpen(msg))           => services.dataplane.open(ctx, msg).await,
        Some(Body::DataChunkMeta(msg))      => services.dataplane.chunk_meta(ctx, msg).await,
        Some(Body::DataCommit(msg))         => services.dataplane.commit(ctx, msg).await,
        Some(Body::ObjectLeaseRenew(msg))   => services.dataplane.renew_lease(ctx, msg).await,
        Some(Body::ResumeRequest(msg))      => services.session.handle_resume(ctx, msg).await,
        Some(Body::NodeHello(msg))          => services.node.handle_hello(ctx, msg).await,
        Some(Body::RegisterCapabilities(m)) => services.node.register(ctx, m).await,
        Some(Body::RemoteInvoke(msg))       => services.node.remote_invoke(ctx, msg).await,
        _ => Err(FapError::unsupported_message()),
    }
}
```

## 结语

FAP-1 的核心判断：

```text
先让 FAP Core 轻、稳、可调试，再逐步叠加高性能与研究能力。

FAP Core Lite:        最小可用协议内核，HTTP JSON + Protobuf + SQLite
FAP Core Hardening:   SemVer + Receipt Multisig + Replay + Idempotency + Hash Chain
FAP Data Stability:   动态背压 + ObjectRef TTL + Lease + Replica + GC
FAP Developer Exp:    FAP-Inspector + Golden Replay + Envelope Decoder + Receipt Verifier
FAP-HP Local:         动态工具注入 + 上下文折叠 + DAG 批量 + 本地调度
FAP-HP Cluster:       分布式能力节点 + 跨节点对象副本 + 远程能力路由
FAP-EXT:              DID/VC + 潜空间 + Gossip + CRDT，默认关闭、编译期可剔除
```

真正超越 VCPToolBox 的前瞻性体现在：

```text
QUIC 原生多路复用 + 零拷贝数据面
Multisig Receipt + Hash Chain + Transparency Log
Mandate 委托链（UCAN/Macaroon 风格）
DAG 批量调用
ObjectRef Lease + 副本 + GC
插件 API 独立 SemVer 双向纳管
编译期剔除的 EXT Profile
版本降级攻击防护
插件运行时三模态（Rust/WASM/Sidecar）
FAP-Inspector 工具链前置
```

方案可按 Phase 0 立即冻结协议，Phase 1 两周交付 Core Lite MVP。
